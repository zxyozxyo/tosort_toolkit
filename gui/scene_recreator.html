Originals in the Scene ZIPs folder are never modified. All repair work happens on copies written to the Output folder. Reference folder files are also never modified — they're copied to a temp location before any inspection. Reference zips are matched against the DAT the same way as the main scan (literal name, release name, or date-stripped release name), then CRC/MD5/SHA1-verified before being trusted as a reference. Each verified reference teaches the tool that release group's actual packer settings (compression level/strategy, entry order) AND its exact set of legitimate entry names — any file in a broken zip absent from every verified reference for that group is treated as junk regardless of its name, catching injected files that pattern-matching alone would miss. Both are tried first — much faster and more reliable than blind searching — before falling back to the general techniques. You can add several reference folders (e.g. specific subfolders) instead of one huge parent folder, to speed up the scan.

Techniques tried in order per file: EOCD comment strip → grow-append truncation → line-ending fix (.nfo/.diz) → junk-file removal → compression variants (STORE + DEFLATE levels 0-9) → faithful rebuild (original headers preserved, searches entry order + ALL compression levels 0-9 + zlib strategy) → heuristic junk removal (strips likely FTP-script adverts/courier tags by name pattern, including folder names) → strip to essentials (last resort: keeps only the largest file + .nfo/.diz, discards everything else regardless of name) → foreign-packer diagnosis. Every structural technique re-runs the full order/compression/strategy search (plus any learned group fingerprints) rather than trying one default rebuild, since the right fix is often a structural change AND a specific compression setting together. Matching stops at the first technique that hits the DAT's target CRC32, then MD5/SHA1 are verified before accepting the fix. If nothing matches but every entry's internal CRC checks out, it's flagged as "likely foreign packer" — content is probably correct but was packed by a tool whose exact compressed bytes can't be reproduced.
Output is organised by DAT release name: output/<game name>/<filename>. Sidecar .nfo/.diz files next to a confirmed .zip are also checked — if the DAT's same release block has a hash entry for them they're verified/fixed and placed alongside the .zip; otherwise they're left untouched in source. Only hash-confirmed files are moved or copied — unresolved files always stay in place.
